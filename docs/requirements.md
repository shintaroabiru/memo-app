# 機能要件定義書（Requirements）

本ドキュメントは、メモアプリの機能要件を定義する。
[project-overview.md](./project-overview.md) と併せて、AI駆動開発・ドキュメント駆動開発の起点として活用する。

---

## 1. 機能スコープ

### フェーズ1（本ドキュメントの主対象）
- メモのCRUD（作成・読取・編集・削除）
- メモの一覧表示
- メモの検索
- タグ管理
- ピン留め（お気に入り）
- ユーザープロフィールの編集

### フェーズ2（別ドキュメントで詳細化）
- ユーザー新規登録
- ログイン/ログアウト等の認証機能

> フェーズ1では認証なしで動作させるが、フェーズ2を見据えてデータモデルには `user_id` を持たせる前提で設計する（詳細は `db-schema.md` で定義）。

## 2. データモデル

### 2.0 文字列フィールドの共通バリデーション規則

すべての**ユーザー入力の文字列フィールド**（`title` / `name` / `display_name` / `bio` 等）に対して、サーバー側で以下の処理を必ず行う。新しいタスクで文字列フィールドを追加する場合も**この規則を必ず踏襲すること**。

1. **前後の空白文字をトリムしてから長さ制約を適用する**
   - 対象: 半角空白・全角空白・タブ (`\t`)・改行 (`\n`, `\r`) など Python の `str.strip()` が除去する全ての文字
   - 実装位置: Pydantic スキーマの `BeforeValidator` で、長さ制約（`min_length` / `max_length`）の評価**前**に適用する
2. **トリム後に空文字列となる入力は `min_length=1` のバリデーションエラー（400 `VALIDATION_ERROR`）として拒否する**
   - すなわち `"   "` / `"\t\n"` のような空白のみの入力は不正
3. **長さ制約はトリム後の文字列に対して評価する**
   - 例: `"  仕事  "` は `"仕事"` に正規化されてから 1〜20 文字の検査を受ける
4. **重複検知もトリム後の値で行う**
   - 空白の有無で同名フィールドが別レコードとして登録されないようにする

> 実装例は [`backend/app/schemas/tag.py`](../backend/app/schemas/tag.py) の `TagName` を参照。`BeforeValidator(_strip_str)` と `Field(min_length=…, max_length=…)` を `Annotated` で合成するパターンを採用している。

### 2.1 Memo（メモ）

| フィールド    | 型              | 必須 | 説明                                     |
|---------------|-----------------|------|------------------------------------------|
| `id`          | UUID            | ✅   | メモの一意識別子                          |
| `user_id`     | UUID            | ✅   | 所有ユーザーID（フェーズ1では仮ユーザー） |
| `title`       | string (1-100)  | ✅   | メモのタイトル                            |
| `body`        | text            | ❌   | メモ本文（**プレーンテキスト**）          |
| `tags`        | string[]        | ❌   | タグ（多対多、後述）                      |
| `is_pinned`   | boolean         | ✅   | ピン留めフラグ（デフォルト: `false`）     |
| `created_at`  | timestamp       | ✅   | 作成日時                                  |
| `updated_at`  | timestamp       | ✅   | 最終更新日時                              |

**バリデーション**
- `title`: 1〜100文字（2.0 の共通規則に従い、前後空白トリム後の長さで判定。空白のみは 400 で拒否）
- `body`: 最大10,000文字（暫定。前後空白はトリムしないが、本文末尾の改行などの扱いは実装時に再検討）
- `tags`: 1メモあたり最大10個、タグ名は1〜20文字（2.0 の共通規則に従う）

### 2.2 Tag（タグ）

| フィールド    | 型              | 必須 | 説明                         |
|---------------|-----------------|------|------------------------------|
| `id`          | UUID            | ✅   | タグの一意識別子              |
| `user_id`     | UUID            | ✅   | 所有ユーザーID                |
| `name`        | string (1-20)   | ✅   | タグ名（ユーザー内ユニーク、2.0 の共通規則を適用） |
| `created_at`  | timestamp       | ✅   | 作成日時                      |

- メモとタグは **多対多**（中間テーブルで管理）

### 2.3 UserProfile（ユーザープロフィール）

| フィールド      | 型              | 必須 | 説明                       |
|-----------------|-----------------|------|----------------------------|
| `id`            | UUID            | ✅   | ユーザーID                  |
| `display_name`  | string (1-50)   | ✅   | 表示名（2.0 の共通規則を適用）              |
| `bio`           | string (0-200)  | ❌   | 自己紹介（2.0 の共通規則を適用、空文字列は許可） |
| `avatar_url`    | string          | ❌   | アバター画像URL（任意。2.0 の共通規則を適用） |
| `created_at`    | timestamp       | ✅   | 作成日時                    |
| `updated_at`    | timestamp       | ✅   | 最終更新日時                |

> フェーズ1では `user_id` を固定値（例: シード投入された仮ユーザー1名）として扱う。

## 3. 機能要件

### 3.1 メモの新規作成

- **入力**: `title`（必須）、`body`、`tags`、`is_pinned`
- **挙動**:
  - 保存時に `id`、`created_at`、`updated_at` を自動付与
  - タグは**既存タグからの選択のみ**（保存時に未登録タグは作成しない）
  - 新規タグが必要な場合は、タグ管理モーダル（3.7参照）で先に作成する
- **画面要件**:
  - タイトル入力欄、本文入力欄（textarea）、タグ選択UI（既存タグからの選択）、ピン留めトグル
  - タグ選択UI内に「タグを新規作成」ボタンを配置し、押下でタグ管理モーダルを開く
- **エラーケース**:
  - タイトル未入力 → バリデーションエラー
  - 本文超過 → バリデーションエラー
  - 存在しないタグIDが指定された → APIエラー（通常UIからは発生しない）

### 3.2 メモの編集

- 既存メモの全フィールドを更新可能
- 更新時に `updated_at` を自動更新
- バリデーションルールは新規作成と同様

### 3.3 メモの削除

- **物理削除**で実装（フェーズ1）
- 削除前に確認ダイアログを表示
- 関連するタグの紐付け（中間テーブル）も削除

> 将来的に「ゴミ箱」機能を追加する可能性があれば論理削除（`deleted_at`）への切り替えを検討。

### 3.4 メモの一覧表示

- **デフォルトソート**: `is_pinned` 降順 → `updated_at` 降順
- **ページング**: 1ページあたり20件（暫定）
- **表示項目**: タイトル、本文の先頭100文字、タグ、更新日時、ピン留めアイコン
- **フィルタ**:
  - タグによる絞り込み（複数選択可、AND条件）
  - ピン留めのみ表示トグル

### 3.5 メモの検索

- **検索対象**: `title` + `body`
- **検索方式**: 部分一致（大文字小文字を区別しない）
- **挙動**:
  - 検索バーへの入力で実行（300msのデバウンス推奨）
  - 検索結果は3.4のソート順を維持
  - 空クエリの場合は通常の一覧を表示
- **将来拡張**: PostgreSQLの全文検索（`tsvector`）への移行を視野に入れる

### 3.6 ピン留め（お気に入り）

- メモ一覧および詳細画面からトグル可能
- ピン留めされたメモは一覧の最上部に表示
- ピン留め数の上限は設けない（暫定）

### 3.7 タグ管理

タグの作成・編集・削除は**専用のモーダル画面**で行う。

- **起動経路**:
  - メモ作成・編集画面のタグ選択UI内「タグを新規作成」ボタン
  - メモ一覧画面のタグフィルタ近くに配置する「タグ管理」ボタン
- **機能**:
  - タグの新規作成（`name`を入力して保存）
  - タグ名の編集（リネーム）
  - タグの削除（紐付くメモからも紐付けを解除）
- **バリデーション**:
  - タグ名は1〜20文字、ユーザー内でユニーク
  - 2.0 の共通規則に従い、前後空白はサーバー側でトリムされる（`"  仕事  "` → `"仕事"`）
  - 空白のみの入力は 400 `VALIDATION_ERROR` で拒否
  - トリム後の値で重複検知するため、空白の有無で別タグにはならない
  - 重複時はエラー表示
- **挙動上の注意**:
  - メモの保存処理では既存タグへの紐付けのみを行い、タグ自体の新規作成は発生しない
  - モーダルでタグを削除した場合、関連する中間テーブルのレコードも削除する

### 3.8 ユーザープロフィールの編集

- `display_name`、`bio`、`avatar_url` を編集可能
- 編集後は即座に反映
- アバター画像は外部URLの指定のみ（アップロード機能はフェーズ2以降）

| 項目         | 内容                                                     |
|--------------|----------------------------------------------------------|
| 対応ブラウザ | モダンブラウザ最新版（Chrome / Edge / Safari / Firefox） |
| レスポンシブ | PC / タブレット / モバイル                                |
| パフォーマンス | 一覧表示で500件程度までは体感ストレスなく動作              |
| 国際化       | 日本語のみ（暫定）                                         |

## 6. API要件（概略）

詳細は `architecture.md` 以降で定義するが、想定エンドポイントは以下。

| メソッド | パス                | 概要                  |
|----------|---------------------|----------------------|
| `GET`    | `/api/v1/memos`        | メモ一覧（検索・フィルタ含む） |
| `POST`   | `/api/v1/memos`        | メモ新規作成          |
| `GET`    | `/api/v1/memos/{id}`   | メモ詳細取得          |
| `PUT`    | `/api/v1/memos/{id}`   | メモ更新              |
| `DELETE` | `/api/v1/memos/{id}`   | メモ削除              |
| `PATCH`  | `/api/v1/memos/{id}/pin` | ピン留めトグル      |
| `GET`    | `/api/v1/tags`         | タグ一覧              |
| `POST`   | `/api/v1/tags`         | タグ新規作成          |
| `PUT`    | `/api/v1/tags/{id}`    | タグ更新（リネーム）  |
| `DELETE` | `/api/v1/tags/{id}`    | タグ削除              |
| `GET`    | `/api/v1/profile`      | プロフィール取得      |
| `PUT`    | `/api/v1/profile`      | プロフィール更新      |
