## 4. 開発ワークフロー（TDD）

このプロジェクトでは **テスト駆動開発（TDD）** を採用します。
**実装より先にテストを書く** を厳守してください。

### 4.1 基本サイクル（Red → Green → Refactor）

```
1. Red    : 失敗するテストを書く
2. Green  : テストが通る最小限のコードを書く
3. Refactor: コードを整理する（テストは通ったまま）
```

### 4.2 機能実装時の手順

新しい機能を実装するときは、以下の順序で進めてください。

1. **要件確認**: `docs/requirements.md` で機能仕様を確認
2. **設計の確認**: `docs/architecture.md` / `docs/db-schema.md` で構造を確認
3. **テスト作成**: 期待される振る舞いをテストとして記述（この時点では失敗する）
4. **テスト実行**: 失敗することを確認（Red）
5. **最小実装**: テストを通す最小限のコードを書く
6. **テスト実行**: テストが通ることを確認（Green）
7. **リファクタ**: 重複排除・命名改善・責務分離（Refactor）
8. **コミット**: 意味のある単位でコミット

### 4.3 テストの粒度方針

| レイヤー                          | テスト種別           | 重点                                   |
| --------------------------------- | -------------------- | -------------------------------------- |
| バックエンド `services/`          | 単体テスト           | ビジネスロジックの正常系・異常系を網羅 |
| バックエンド `repositories/`      | 単体テスト           | DB操作の正しさ（テストDBで実施）       |
| バックエンド `api/`               | APIテスト            | エンドポイント単位の入出力検証         |
| フロント `features/` 内のロジック | 単体テスト           | hooks / 純粋関数の振る舞い             |
| フロント コンポーネント           | コンポーネントテスト | 主要な操作と表示（最小限でよい）       |

### 4.4 テスト実行コマンド（暫定）

```bash
# バックエンド（pytest）
cd backend && pytest

# フロント（Vitest）
cd frontend && npm test
```

> 実際のコマンドはプロジェクト初期化時に確定する。

### 4.5 Lint / Formatter

| 対象         | ツール                | コマンド（暫定）                                |
| ------------ | --------------------- | ----------------------------------------------- |
| バックエンド | **Ruff**              | `cd backend && ruff check . && ruff format .`   |
| フロント     | **ESLint + Prettier** | `cd frontend && npm run lint && npm run format` |

- コミット前にローカルでLint/Formatを実行する
- 将来的に pre-commit / lint-staged の導入を検討

## 5. コーディング規約

### 5.1 共通ルール

- **小さく作る**: 1関数・1コンポーネントは責務を1つに絞る
- **早期リターン**: ネストを深くしない
- **マジックナンバー禁止**: 定数化して命名する
- **コメントは「なぜ」を書く**: 「何を」はコードで表現する

### 5.2 TypeScript / フロント

- **`any` 禁止**（やむを得ない場合は `unknown` を使い、絞り込む）
- **型は明示的に**: 関数の引数・戻り値は型注釈を付ける
- **Zodスキーマから型を生成**: `z.infer<typeof schema>` を活用
- **コンポーネントは関数コンポーネント + Hooks**
- **`features/{機能名}/` に閉じる**: 機能固有のコンポーネント・hooks・型はその機能配下に置く
- **`components/ui/` には汎用UIのみ**: 2機能以上で使うものに限る

### 5.3 Python / バックエンド

- **型ヒント必須**: 関数の引数・戻り値に型を付ける
- **`Any` を避ける**: 適切な型で表現する
- **Pydantic スキーマと SQLAlchemy モデルを分離する**: APIの入出力とDB表現を混ぜない
- **レイヤー責務を守る**:
  - `api/` はHTTPだけを扱う
  - `services/` がビジネスロジック
  - `repositories/` がDB操作
  - `models/` はテーブル定義のみ
- **依存性注入**: FastAPIの `Depends` を活用してテスタブルに保つ

### 5.4 命名規約

| 対象                 | 規約                                       |
| -------------------- | ------------------------------------------ |
| React コンポーネント | PascalCase（`MemoCard.tsx`）               |
| カスタムフック       | `use` プレフィックス（`useMemoList`）      |
| Zustand ストア       | `xxxStore`（`memoStore.ts`）               |
| Python クラス        | PascalCase                                 |
| Python 関数・変数    | snake_case                                 |
| DBテーブル名         | snake_case・複数形（`memos`, `memo_tags`） |
| APIパス              | 複数形を採用（`/api/v1/memos`）            |

## 6. ディレクトリ構造の遵守

`docs/architecture.md` で定義された構造に従ってください。

### フロント（重要ルール）

- `app/` はルーティングのみ。ロジックは `features/` に置く
- `app/api/` は BFF として FastAPI を呼ぶだけ。ビジネスロジックを書かない
- 機能固有のコンポーネントは `features/{機能名}/components/` に置く
- `components/ui/` は汎用UIのみ

### バックエンド（重要ルール）

- `api/v1/` 配下にエンドポイントを配置
- ビジネスロジックは `services/` に集約
- DB操作は `repositories/` に閉じる
- `models/` はSQLAlchemyのテーブル定義のみ
- `schemas/` はPydanticの入出力スキーマのみ

## 7. Git運用

### 7.1 ブランチ戦略

- **`main`**: 常に動作する状態を保つ。直接コミットは禁止
- **`feature/{機能名}`**: 機能単位で切る
  - 例: `feature/memo-crud`, `feature/tag-management`, `feature/search`
- **`fix/{修正内容}`**: バグ修正
- **`docs/{ドキュメント名}`**: ドキュメントのみの変更
- **`refactor/{対象}`**: 機能変更を伴わないリファクタ

### 7.2 コミットメッセージ

**Conventional Commits** に準拠してください。

```
<type>: <subject>

<body（任意）>
```

**type 一覧**

| type       | 用途                                   |
| ---------- | -------------------------------------- |
| `feat`     | 新機能の追加                           |
| `fix`      | バグ修正                               |
| `docs`     | ドキュメントのみの変更                 |
| `style`    | フォーマット・空白等（動作に影響なし） |
| `refactor` | リファクタリング                       |
| `test`     | テストの追加・修正                     |
| `chore`    | ビルド・ツール・依存関係の更新         |

**例**

```
feat: メモの新規作成APIを追加

- POST /api/v1/memos エンドポイントを実装
- バリデーションスキーマを追加
- 単体テストを追加
```

### 7.3 コミットの粒度

- **1コミット = 1論理変更**
- テストとそれを通す実装は **同一コミット** にまとめてよい
- リファクタは別コミットに分ける
- WIP コミットは push 前に整理する

## 8. ドキュメント駆動の運用

このプロジェクトは **ドキュメント駆動開発** で進めます。

### 8.1 ドキュメントが先

- 仕様変更・新機能追加の際は、**先に `docs/` 配下のドキュメントを更新**してください
- 実装はドキュメントを参照しながら進める
- 「コードとドキュメントが食い違っているとき、ドキュメントを正とする」

### 8.2 ドキュメント更新タイミング

| 状況                         | 更新対象                  |
| ---------------------------- | ------------------------- |
| 機能仕様の変更               | `requirements.md`         |
| アーキテクチャの変更         | `architecture.md`         |
| DBスキーマの変更             | `db-schema.md`            |
| API仕様の変更                | `api-spec.md`             |
| ワークフロー・規約の変更     | `CLAUDE.md`（本ファイル） |
| プロジェクト全体の方針変更   | `project-overview.md`     |
| UIデザイン・画面モックの変更 | `docs/design/design.pen`  |

### 8.3 確認を取るタイミング

以下のケースでは、**実装を進める前に必ずユーザーに確認**してください。

- 既存ドキュメントに記載のない仕様判断が必要なとき
- 技術スタックを追加・変更したいとき
- ディレクトリ構造を変更したいとき
- ライブラリを新規導入したいとき
- 破壊的な変更（DBスキーマ変更、API仕様変更など）が必要なとき

## 9. Claude Code への作業指示パターン

ユーザーから依頼を受けたときの推奨ワークフローです。

### 9.1 機能実装の依頼を受けたとき

1. 関連ドキュメント（`requirements.md` / `architecture.md` / `db-schema.md`）を読む
2. **実装計画を提示**してユーザーの確認を得る
   - 影響範囲（変更ファイル）
   - 実装ステップ
   - テスト方針
3. ユーザー承認後、TDDで実装
4. 完了後、変更点をサマリ報告

### 9.2 不明点があったとき

- **勝手に判断しない**
- 仕様の解釈に複数の可能性がある場合は、選択肢を提示して確認
- ドキュメントに記載がない場合は「ドキュメント更新が必要では？」と提案

### 9.3 エラー対応

- スタックトレースを読み、原因を推定して説明
- 修正方針を提示してから実装
- 場当たり的な修正（例: `try-except` で握りつぶす等）はしない

---

## 10. 禁止事項

- ❌ ドキュメントを読まずに実装を始める
- ❌ テストなしで機能コードを書く（TDDの原則違反）
- ❌ 仕様の自己判断（不明点は必ず確認）
- ❌ 技術スタックの勝手な追加・変更
- ❌ `any` / `Any` 型の濫用
- ❌ `main` ブランチへの直接コミット
- ❌ レイヤー責務の混在（例: `api/` にビジネスロジックを書く）
- ❌ 機密情報のコミット（`.env` ファイル等は `.gitignore` に追加）
